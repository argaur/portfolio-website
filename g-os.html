Runs before /rubric, for every project — Blueprint and Builder OS both consume a PRD.md, so writing it can't live inside either framework. /prd-create is a 6-stage state machine that produces PRD.md with a Provenance block, which /rubric then requires and cites for each of its 5 answers.

Ground rules (apply to every stage)
Gaurav owns the problemClaude never invents or assumes the idea, pain point, or problem statement — only surfaces, refines, and pressure-tests what Gaurav gives it.
2–3 options, alwaysEvery decision point offers real alternatives with tradeoffs. Never a single unilateral call.
Research stays separatePRIMARY_RESEARCH.md and SECONDARY_RESEARCH.md are never merged, and never get folded into the PRD as first-draft prose — the PRD cites them.
Broad before narrow, alwaysNever assume a problem statement and research backward to justify it. Divergence (Stage B) happens before any research question is written.
The 6 stages
A · Spark→ B · Diverge→ ◆ git-lock→ C · Research→ D · Narrow→ E · Draft→ F · Grill + Seal
A
Spark
Gaurav's raw idea, captured verbatim — no reframing.
Gaurav dumps the raw idea/pain point in whatever form it comes. Claude asks clarifying questions one at a time — never reframes or narrows. Captured near-verbatim into SPARK.md. Everything defaults to 🔵 hypothesis.
B
Problem-space map (diverge)
≥3 framings incl. a mandatory anti-thesis — no picking yet.
Produces ≥3 distinct problem framings, including one mandatory anti-thesis framing — the version where this isn't worth solving, or the pain is adjacent. Presented as options with tradeoffs. Gaurav does not pick yet — output is a research agenda, not a decision.
◆ Gate: "Framings approved."
C
Research bias-lock
Questions frozen via git commit before narrowing is allowed.
Research questions are derived from all Stage B framings and committed to git before any narrowing is allowed — this is the actual lock, not an honor system; questions can't be quietly rewritten once the desired answer is known. Two files, never merged: PRIMARY_RESEARCH.md (interview scripts + transcripts — no 🟢 tag anywhere without a quotable line here) and SECONDARY_RESEARCH.md (market/competitor/community scan, formatted as a Steal List). Both require an Evidence Against section with ≥3 entries, or an explicit "found nothing" log.
Fast pathexperiment-brief.md gets a ~30-min secondary-only sweep instead of the full cycle. An all-🔵/🟡 experiment brief is the expected shape, not a failure.
D
Narrow (converge)
2–3 candidate statements with evidence for/against — Gaurav picks.
Problem clustering → P1→P2→P3→core-failure chain → 2–3 candidate narrowed problem statements, each with supporting and contradicting evidence cited by file anchor, plus "what evidence would flip this." Gaurav picks and finalizes the wording — Claude never authors the problem statement, only the evidence dossier behind each candidate.
◆ Gate: "Problem statement finalized."
E
Template select + draft
Pick from 4 templates by stakes × type, draft with cited claims.
Template picked via the decision matrix below. Draft carries every claim with a confidence tag plus a reference into the research files — the PRD cites research, it doesn't restate it. Also produces (required): every success criterion rewritten into falsifiable form, and the Prior Art & Steal List carried forward for Blueprint Phase 0.
F
Grill + seal
grill-me on the discovery chain; sealed into PRD.md with Provenance.
grill-me targets the discovery chain specifically: "which Stage B framing did the research actually kill?" — if the answer is "none," that's surfaced as a smell, not silently passed. Survivors seal into root PRD.md with a Provenance block (template used, stakes classification, research file refs, grill date).
◆ Gate: "PRD approved." → next: /rubric, now required to cite PRD.md.

Phases
P0
PRD Intake Provenance short-circuit
Read PRD → 5-bullet read-back → falsifiable success criteria → ambiguity list → Prior Art steal list.
Steelman Gate: state the polarizing claim → sharpest counter (2 sentences) → defense (2 sentences). If defense doesn't hold, revisit scope before Phase 1.
Provenance short-circuitIf PRD.md already carries a Provenance block (written by /prd-create), steps 2–5 are read directly from the PRD instead of freshly generated — Phase 0 becomes a confirmation gate, not a repeat of the work.
◆ Gate: "Read-back approved."
P1
Solution Stage
POT-format interview (Proposal → Options → Tell me). 8 topics in order: core journey, MVP features, non-goals, tech shape, 3rd-party services, metrics, recruiter signal, risk register. One session, max 10 questions.
◆ Gate: "Solution approved."
P2
Design Named Reader added
7-stage Design Framework (see Section 03). Required artifact from Stage 1: Named Reader — one specific real person this product must land for. Stored in brand-guide.md.
◆ Gate: "Spec approved."
P3
Implementation Plan
Slice 0: walking skeleton (frontend shell + /health + DB + PostHog + Sentry + /docs stub). Slices ordered by risk. Each slice: one capability + tests + analytics + HOW_TO_USE.md section.
◆ Gate: "Plan approved."
P4
Development
Per-slice: Red (failing tests) → Green → Refactor → Instrument → Document → Review → Smoke-run → Commit. Between slices: 5-line PROGRESS.md + /compact.
◆ Gate: All tests green. HOW_TO_USE.md complete.
P5
Testing
Layer 0: Security pass (IDOR, authz, input sanitization). Layer 1: Unit + Integration + E2E. Layer 2: 8-category edge case checklist. Layer 3: Acceptance criteria.
◆ Gate: CI green · zero P0/P1 bugs · acceptance criteria walked.
P6
Deployment
13-item checklist: env vars, secrets, /health, DB migrations, PostHog, Sentry, HTTPS, CORS, rate limits, backups, CI green, README, CASE_STUDY.md.
◆ Gate: Definition of Shipped — 6 conditions confirmed.
Reclassification triggers

Auth / payments / 3+ entities / real users appear → promote to Blueprint.
Kill criterion fires or 2+ weeks stalled → demote to Builder OS.
Both require Gaurav confirmation + CLAUDE.md update.

For solo MVPs: 0–1 rubric questions. Speed is the non-negotiable. PRD.md now arrives pre-made via /prd-create with a Provenance block — don't hand-fill a blank template. USER_FLOWS.md is extracted verbatim from the PRD's Product Flows section during bootstrap, not written separately.

Data Model
All entities, relationships, every state (empty / loading / error / populated). Mandatory before any visual work. Output: DATA_MODEL.md.

S2
Interview
One question at a time. Step 0: Purpose Statement. Track A: design language derived from purpose (visual feel, references, fonts, negative constraint "what would make this look AI-generated?"). Track B: per-panel intent with real copy.
S3
Wireframes now a decision, not a fixed tool
Excalidraw (default) or Figma — decided by a 3-question matrix.
Layout only. Component map + key screens + unhappy states (empty/error) + end-to-end flow + friction annotation (one word per transition: confusion / hesitation / exit / scroll / load / skip).

S4
Design System now a decision, not a fixed tool
Claude Design (default) or Stitch — decided by a 4-question matrix.
Every token carries a value + usage rule. Component showcase. Inevitability test on the full set. Output: tailwind.config + CSS vars + components.json + brand-guide.md.
#QuestionYes →
1Is code the final medium (shadcn/Tailwind)?Claude Design
2Does the design language fit Stitch's font enum?No → Claude Design (forced)

(3 or 4) AND 2 → Stitch first, then still emit tokens as code. Otherwise → Claude Design (default). Stitch isn't a safer bet against generic output — the anti-slop guards apply to both paths equally.

Spec Doc
