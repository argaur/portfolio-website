The Problem

Generic PM Content Doesn't Tell You Where You Stand

Aspiring and transitioning PMs face the same five gaps: no portfolio to prove capability, no referral network into PM roles, content consumed but never converted into applied skill, no structured feedback outside real interviews, and PM education written by PMs for PMs that alienates career switchers.

PM Pathfinder addresses this with a 10-question diagnostic that places each user into one of 6 PM archetypes (Builder, Architect, Storyteller, Advocate, Operator, Strategist) and a roadmap targeted at their specific gap — not a generic course. The build challenge I set myself: ship a production RAG pipeline end-to-end, including deploying and owning the vector database, rather than wrapping a managed RAG API.

Architecture

Retrieval Pipeline, End to End

Source content (archetype playbooks, a "builder's bible" of PM frameworks) is chunked and embedded with Google's gemini-embedding-001 model into 3072-dimension vectors, then written to a document_chunks table in Supabase Postgres with the pgvector extension enabled. An IVFFlat index (lists=50, cosine ops) keeps similarity search fast as the corpus grows.

At chat time, the user's message is embedded with the same model, then matched against document_chunks through a match_chunks() Postgres function: a plpgsql RPC that ranks by cosine distance and optionally filters to the user's diagnosed archetype (falling back to the shared "builder's bible" source when no archetype match exists). The top 4 chunks are stitched into the system prompt before the generation call — classic retrieve-then-generate, not a fine-tune or a static FAQ.

Why Not Just Call the Gemini SDK

  • Raw REST over the SDK — the Node Gemini SDK hit Vercel Edge/serverless runtime incompatibilities; hand-rolled fetch calls to the REST endpoints removed that dependency entirely.
  • SSE streaming parsed manually — the chat route reads the streamGenerateContent response as a raw byte stream, splits on data: lines, and re-streams plain text to the client for a typing effect with no extra latency.
  • 429 backoff built in — embedding and generation calls retry with exponential backoff (1s/2s/4s) before failing, since the free-tier Gemini API rate-limits aggressively under burst traffic.
Build Decisions

Key Design Choices

Why deploy my own vector DB instead of Pinecone or a managed RAG service? Supabase was already the auth and data layer for the app. Enabling pgvector on the same Postgres instance meant one fewer service to provision, one fewer API key to rotate, and row-level security applied consistently with the rest of the schema — while forcing me to actually understand chunking, embedding dimensionality, index tuning, and similarity search instead of treating retrieval as a black box.

Why archetype-aware filtering in the match function? A Builder archetype and a Storyteller archetype need different framing for the same PM concept. The match_chunks() function accepts an optional archetype filter so retrieved context is personalised, but always allows the shared "builder's bible" through so general PM questions never come back empty.

Why service-role keys only in the API route? document_chunks has RLS enabled with no public read policy — only the server-side Supabase client (service role key, never shipped to the browser) can query it. The chat route also gates on an authenticated user before it will spend an embedding call.
