The Problem

Context-Switching Was the Tax

Every personal workflow lived in a different app. Saving an idea meant opening Notion. Logging a thought meant opening Obsidian. Checking today's calendar meant opening Google Calendar. Writing a LinkedIn post meant opening a browser, a notes app, and an image tool in sequence.

The cost wasn’t the tasks themselves — it was the friction of switching between 6+ apps to do anything meaningful. I needed one interface that could reach all of them.

Architecture

One Entry Point, Any System

A single bot listener runs on Oracle VM (systemd, always-on). Every Telegram message goes through a keyword router in bot/router.py. Matched commands dispatch to typed handlers. Unmatched messages fall through to Claude Haiku — Tavily-grounded for factual queries.

Architecture flow: 
You → Telegram (mobile, anywhere)
  ↓
Oracle VM — Bot Listener (systemd, always-on)
  ↓
Keyword Router (bot/router.py)
  ↓ no match
Freetext catch-all → Claude Haiku (+ Tavily grounding)
  ↓
Handler → GWS CLI / Claude API / Notion / Obsidian / GitHub / YouTube
  ↓
Action executed + confirmation → Telegram

Google Workspace operations (Gmail, Calendar, Drive, Tasks, Contacts) run through GWS CLI as a subprocess using system Python — keeping its dependencies isolated from the bot’s virtualenv. All async Telegram handlers wrap sync I/O with asyncio.to_thread(). State is tracked in state.json with filelock and atomic writes.

Commands

What It Does

Content & LinkedIn Pipeline

  • #idea — Run Idea Refiner → save to Notion Post Ideas DB
  • #post — Generate LinkedIn draft + DALL-E 3 cover image → Notion Posts DB
  • #done — Mark latest draft as published
  • #scan — Gmail newsletter digest, relevance-scored via Claude
  • #jd [text or file] — Generate interview prep guide from job description (.txt / .pdf / .docx)
  • URL drop — Auto-summarise → Notion Saved Links DB (YouTube URLs get title + channel metadata)

Information Retrieval

  • #links [N] [#tag] — Retrieve saved links from Notion with optional count + tag filter
  • #cal / #cal today / #cal week — Google Calendar events (attendees enriched via Contacts)
  • #drive [query] — Search Google Drive files
  • #drive recent — Last 3 modified Drive files
  • #gh — Open PRs + recent commits (GitHub)
  • #rss add <url> <type> / #rss list — Manage RSS and Reddit feed sources

Task & Personal Log

  • #task <project> <desc> — Add task to Notion Tasks DB + Obsidian
  • #task done <partial> / #task list [project] — Manage tasks
  • #capture [text] — Timestamped bullet → Obsidian Life Logs + Notion Captures DB
  • #note [text] — Daily note → Obsidian Life Logs/Notes/YYYY-MM-DD.md
  • #journal [text] — Journal entry → Obsidian dated folder
  • #think [text] — Structured thinking with persistent memory (past decisions injected as context)
  • #career <entry> — Append dated entry to Obsidian career log

Utilities

  • #reauth — Re-authorise Google Workspace via device flow; no SSH required
  • #status — Bot service state and uptime
  • /help or #help — Dynamic command list
  • Anything else — Natural language → Claude Haiku (Tavily-grounded for factual queries)
Automations

5 Daily Cron Jobs

Beyond on-demand commands, 5 scheduled jobs run automatically every day on the Oracle VM:

Time (IST)JobOutput
06:30GWS auth checkTelegram alert if token near expiry (2h warning before newsletter)
08:00Newsletter pipelineGmail → Obsidian → idea miner → Notion (silent)
08:30Morning briefTelegram: calendar · today’s focus · tasks · GitHub · news
09:00Content digestTelegram: newsletters + RSS picks with relevance scores

The newsletter → LinkedIn pipeline is fully automated end-to-end: Gmail newsletters land in Obsidian at 08:00, the idea miner scores and saves them to Notion, and the evening job picks from the queue to generate a post + cover image for review before publishing.

Build Decisions

Key Technical Choices

GWS CLI as subprocess, not a library. Google Workspace operations run through GWS CLI using system Python 3 — not the bot’s virtualenv. This keeps dependency graphs separate and avoids conflicts between Google API client libraries and the bot’s async environment.

Anthropic SDK is sync — never await it. All async Telegram handlers that call Claude wrap the sync calls with asyncio.to_thread(). This was a silent failure mode early on; the rule is now enforced throughout the codebase.

Atomic state writes with filelock. state.json tracks pipeline state (last newsletter read, last idea mine). Two cron jobs writing simultaneously caused corrupt state early in development. Every write now goes through integrations/state.py with a filelock and atomic rename.

CI/CD via GitHub Actions. git push origin main triggers a workflow that deploys to Oracle VM in ~30 seconds. No manual SSH for deployments.

Status

Fully Operational

All 28+ commands shipped and in daily use as of May 2026. The 5 cron automations have run without manual intervention since launch. One known limitation: GCP OAuth app in Testing mode caps token lifetime at 7 days — mitigated with the 06:30 auth-check alert and the #reauth device-flow instruction card.

Decided Against

  • Spotify integration — requires Premium subscription; parked
  • Product launch feature — deferred; warrants a separate codebase
  • OpenClaw — OpenAI backing creates Claude compatibility risk; current bot covers all flows natively
